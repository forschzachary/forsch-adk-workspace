# Networking: collapse four paths to one, keep the simple auth you already have

> **Second-opinion plan.** A prior plan in this file proposed "keep cloudflared
> tunnel + add Cloudflare Access, delete the rest." This plan corrects two things
> in that proposal and converges on a simpler answer: **the auth you asked for
> already exists (`X-Graph-Secret`); the real pain is four overlapping network
> paths, one of which is broken right now. Fix the network, leave auth alone.**

## What you're building (recap, for grounding)

`live-agent-graph` is a **control surface**: a force-graph that reflects live
agent state (ADK bridge + LiteLLM + Authsome on a Hetzner VPS) and lets an
operator spawn/wire/promote agents. `serve.py` is the backend. The graph is
viewed standalone at `graph.forschfrontiers.com` and embedded inside Frappe CRM.

## Evidence: the current request flow is four paths to one server

| # | Path | Evidence |
|---|---|---|

| B | CRM-embedded browser → CRM `graph_embed` proxy → `serve.py` (CRM **injects `X-Graph-Secret`** server-side) | `index.html:649-657` (`apiUrl()`), `.hermes/plans/...:740` |

**Tailscale** (`infra.yaml:39`, host `hubert-cloud-sp6`) is operator-only
(SSH target `root@100.120.21.13`). Not in any browser path. Healthy. Untouched.

## Diagnosis — why "a lot of tunneling" is accurate, and why "simple auth" is half the answer

1. **Auth is already simple and already works.** `serve.py:1197-1239` enforces
   `X-Graph-Secret` on every mutating endpoint; reads (`/pulse`, `/clusters`,
   `/manifest`, `/models`) are intentionally unauthed; CORS is pinned to
   `crm.forschfrontiers.com` (`serve.py:1554`). The 2026-06-26 Frappe CSRF
   cleanup (`7189f5a`) already collapsed this to one layer. **There is nothing
   for "spin up simple auth" to add — it's done.**
2. **The pain is the network layer in front of auth.** Four paths, each can
   break independently, one already has (the 502/Cloudflare 1101 on the live
   URL, `handoff 2026-06-27:68,76`). The Worker is currently **undeployed**
   (Cloudflare OAuth expired, `handoff:62`), so the live URL is broken at the
   edge regardless of what `serve.py` does.
3. **The public-IP plaintext hop is a real security regression hiding in the
   "tunneling."** `cloudflare-worker.js:10` fetches `http://87.99.149.222:8888`
   — unencrypted, over the public internet, to a raw IP. That is *not* a
   tunnel; it's an open HTTP origin pull. Any "fix" that preserves this hop
   preserves the regression.
4. **The CRM `graph_embed` path exists only to inject the shared secret.**
   `.hermes/plans/...:740` confirms it. Once the edge authenticates (or once
   standalone access is the primary path), this whole path is dead weight —
   and `HANDOFF-2026-06-26-agent-focus-zoom.md:134` already flags the CRM
   Agent Builder page as likely-retirable.

## The critical unknown (resolves before implementation)

**Is `cloudflared` actually running as a live daemon on the Hetzner box?**

The prior plan *assumed* yes (`...witty-rocket.md:23-26`) and built its
"keep the tunnel" strategy on that assumption. The evidence contradicts a
clean cloudflared setup: the Worker fetches a **public IP over plaintext
HTTP**, which a properly-configured cloudflared tunnel would not require (the
Worker would fetch `http://localhost:8888` via the tunnel's internal route).
`shared/infra.yaml:43-46` lists the tunnel only as **descriptive metadata**
(the file header, line 3, says "not live config"). Nothing in the repo
confirms a live `cloudflared` process.

**The current 502/1101 is consistent with no bridge existing at all** — not
just a port mismatch. So the prior plan's central assumption is unverified
and may be false.

**Check on the box:** `systemctl status cloudflared` (or `ps aux | grep cloudflared`).

## Recommendation: collapse to ONE path, keep `X-Graph-Secret`, defer Access

Both branches delete the same three things (Worker, CRM `graph_embed`
secret-injection, separate `chat.forschfrontiers.com` hostname) and keep auth
exactly as-is. The only difference is what replaces the broken edge hop.

### Branch 1 — IF cloudflared IS running: keep the tunnel, fix its config

- Delete `cloudflare-worker.js` (superseded — the tunnel handles TLS + routing).
- Point the existing cloudflared tunnel's ingress at `graph.forschfrontiers.com`
  → `127.0.0.1:8888`. Route `/chat/*` → `127.0.0.1:8800` through the **same**
  tunnel (kills the separate chat hostname, path D).
- Move `chat_token` injection (worker.js:13-14) into `serve.py` (≈5 lines) so
  the Worker's only feature survives its deletion.
- Delete the CRM `graph_embed` secret-injection path (`apiUrl()` at
  `index.html:652-657` + the CRM-side `?path=` handler). The CRM, if it still
  embeds the graph, just iframes `graph.forschfrontiers.com` and the user
  authenticates the same way a standalone user does.
- `X-Graph-Secret` stays as the sole auth. No Access. No new vendor surface.
- Net: one tunnel, one hostname, one auth. Zero new daemons.

### Branch 2 — IF cloudflared is NOT running: put Caddy on the box

- Install Caddy on the Hetzner box. One ~15-line Caddyfile:
  `graph.forschfrontiers.com { reverse_proxy 127.0.0.1:8888 }` — automatic
  Let's Encrypt TLS, no cert management, no Worker, no tunnel daemon.
- Keep `serve.py` bound to `127.0.0.1` (Caddy proxies to it). The box opens
  **one** public port (443); LiteLLM `:4000`, Authsome `:7998`, ADK-bridge
  `:8800` stay localhost-only. Blast radius = one port.
- Same deletions as Branch 1 (Worker, CRM embed secret-injection, chat
  hostname folded into a Caddy route `/chat/*` → `127.0.0.1:8800`).
- Move `chat_token` injection into `serve.py` (same ≈5 lines).
- Keep Cloudflare as an **orange-cloud proxy** in front of Caddy (DNS
  orange-cloud → box:443) to retain DDoS/WAF. You lose nothing of the current
  edge protection and gain a single, on-box, debuggable origin over SSH/Tailscale.
- `X-Graph-Secret` stays as the sole auth. No Access.
- Net: one Caddyfile replaces Worker + tunnel + mystery bridge + plaintext hop.

### Why both branches defer Cloudflare Access (divergence from the prior plan)

The prior plan recommended adding Cloudflare Access (Zero Trust) for real
per-user identity. That is a *correct* future improvement — but it is **not**
"simple auth," it is a new vendor surface with its own policy/admin burden, and
the user explicitly asked for simple. `X-Graph-Secret` already satisfies
"simple auth" and already works. Adding Access in this pass would violate the
stated goal. Defer it to a dedicated identity pass once the network is collapsed.
